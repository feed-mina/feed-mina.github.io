PROBLEM × SOLUTION

냉장고는 가득한데,
오늘 뭐 먹을지는 늘 어렵다.

PROBLEM
반복되는 식자재 낭비와 식단 고민
  • 유통기한이 임박한 재료를 자꾸 잊는다
  • 레시피 검색은 많은데, 지금 내 냉장고로는 만들 수 없다
  • 알러지·저염·고단백 같은 개인 조건은 반영되지 않는다
  • 부족한 재료를 사려면 또 따로 검색해야 한다
SOLUTION
"냉장고를 알고 있는" 한 개의 플랫폼
  • 📸 사진 1장으로 냉장고 재료 자동 등록 (YOLO + OCR)
  • 🍳 보유 재료 + 의미 유사도 + 사용자 조건 = 하이브리드 추천
  • 💬 "왜 이 레시피?"를 LLM이 한국어로 설명
  • 🛒 부족한 재료는 쿠팡/네이버/11번가 최저가로 즉시 연결

AI · MODEL 02

PaddleOCR
포장 식품 텍스트 인식

PROBLEM
"라면 한 봉지의 라벨에서 진짜 상품명만 뽑기"
포장 식품 사진에는 영양정보, 인증마크, 가격 라벨, 광고 문구가 뒤섞여 있어 단순 OCR로는 노이즈가 너무 많음.
SOLUTION · packagedFoodOcr.py
PaddleOCR + 정규식 노이즈 필터
  • 한국어 + 영문 detection / recognition
  • 제거: 영양정보(g, kcal, %), 인증마크, 가격, 바코드 텍스트
  • 유지: 길이 2~20자 한글 중심 상품명 후보
  • 상위 N개 후보 중 신뢰도 + 길이 가중치로 선정
CONNECTS TO
🥛 라면 · 우유 · 소스 자동 등록
YOLO 라우터가 packaged_food 판정 시 이 모델로 전달.

DESIGN PRINCIPLES

왜 Redis가
중심에 있는가?

REDIS
Single Source of Truth
📨 이메일 토큰 (24h)
🔑 Refresh 세션
⛔ JWT 블랙리스트
🛒 쇼핑 가격 캐시 (1h)
🔄 토큰 로테이션
📊 MDC 요청 트레이스
01
HttpOnly + Secure 쿠키
JWT를 JavaScript가 읽을 수 없게 차단 → XSS 토큰 탈취 방어. JwtProvider#createTokenCookie에서 일괄 관리.
02
Redis Single Source of Truth
세션, 블랙리스트, 이메일 토큰, 가격 캐시 — 모든 시한부 상태가 한 곳에. TTL로 자동 정리.
03
DB + Redis 이중 저장 (전환기)
기존 DB 토큰 컬럼과 Redis 토큰을 병행 → 무중단으로 인프라 마이그레이션.
04
이커머스 멀티 소싱
쿠팡 · 네이버 · 11번가 — 한 곳이 실패해도 폴백. 어댑터 패턴으로 통일된 DTO.
05
글로벌 일관 응답
ApiResponse{success, code, message, data} + ErrorCode enum — 인증/쇼핑/도메인 모두 동일 포맷.
06
axios 인터셉터 자동 갱신
401 응답 → /refresh 시도 → 실패 시 auth:unauthorized 이벤트 → AuthContext 강제 로그아웃.

                                          │ (HttpOnly cookie)
BE  AuthController.login()        ◀┘
       ├─ AuthService.authenticate()      // BCrypt
       ├─ if (!emailVerified) ─▶ EMAIL_NOT_VERIFIED
       ├─ JwtProvider.createAccessToken(loginId)
       ├─ JwtProvider.createRefreshToken(loginId)
       ├─ authService.createSession()
       │      └─ Redis SET session:{loginId}
       └─ Set-Cookie HttpOnly · Secure · SameSite=Lax

FE  AuthContext.login()
       ├─ getMeApi()                // 닉네임 조회
       ├─ sessionStorage.setItem("authUser", ...)
       └─ router.replace("/dashboard")
// 로그아웃 — POST /api/v1/auth/logout

FE  LogoutButton onClick
       ├─ logoutApi()
       └─ AuthContext.logout()

BE  AuthController.logout()
       ├─ resolveTokenFromCookie("accessToken")
       ├─ resolveTokenFromCookie("refreshToken")
       ├─ getRemainingMs(accessToken)   // 남은 TTL
       ├─ authService.invalidateSession()
       │      ├─ Redis SET blacklist:{jwt}
       │      │   ⏱ TTL = 토큰 잔여시간 (자동 정리)
       │      └─ Redis DEL session:{loginId}
       └─ Set-Cookie Max-Age=0 (즉시 삭제)

FE  sessionStorage.removeItem("authUser")
       └─ window.location.href = "/"

🔧 TROUBLESHOOTING
Next.js 캐싱 ↔ Redis 세션 불일치
증상: 리포트 데이터 캐싱 과정에서 세션 ID 헤더가 오염되거나 세션 연장 요청이 차단 → Redis 세션 강제 만료 → 사용자 강제 로그아웃.
해결: 캐싱에 저장되던 정보를 별도 테이블로 분리 PostgreSQL DB로 이전. 세션과 무관한 도메인 데이터를 헤더와 디커플링.
예정 1: 외부 쇼핑 API 레이트 리밋 → Redis 1h 캐시 + 키워드 디바운스
